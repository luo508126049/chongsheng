# 《重生一亿次》微信纯文本交互 AI 小游戏 — 可交付研发&内容团队 PRD v1.0

> 目标：研发可按接口/数据结构直接开工；内容团队可按“世界观圣经 + 事件卡 + 天赋/成就库 + QA规范”批量生产与验收。

------

## 0. 文档信息

- 版本：v1.0（MVP可上线）
- 端：微信小程序（推荐）/服务号会话（可复用同一后端）
- 交互：聊天流纯文本 + 结构化快捷选项（按钮）+ 少量抽屉面板（状态/背包/关系）
- 核心闭环：开局 → 多轮事件 → 死亡/结局 → 结算 → 天赋选择 → 重生再开局

## 2. 关键用户故事（验收口径）

1. 新用户：进入后 2 分钟内完成“首次死亡→结算→选择天赋→第二次开局”
2. 轻度用户：离开小程序后 24h 内回来能一键“继续本局”
3. 重度用户：连续重生 10 次，天赋与成就能持续驱动，不出现剧情明显自相矛盾或卡死

------

## 3. 页面与交互规格（研发用）

> 小程序页面建议：`pages/home`、`pages/chat`、`pages/settlement`、`pages/talent`、`pages/atlas`、`pages/achievements`、`pages/settings`

### 3.1 首页 Home

**模块**

- 主按钮：开始重生 / 继续本局（若存在 `active_run`）
- 入口：天赋图鉴、成就、设置
- 快捷信息：累计重生次数、已解锁天赋数、记忆碎片余额

**交互**

- 点击开始重生：创建 run（或先进入世界观选择，MVP可默认唯一世界观）
- 点击继续：拉取 run state + chat history 摘要，进入 Chat

**异常**

- active_run 丢失：提示“存档异常，已为你开启新人生”，自动 new run

------

### 3.2 主对话 Chat（核心）

**布局（聊天流）**

- 顶部：世界观名 + 第X次重生 + 当前年龄（可选）
- 中部：消息列表（AI/用户）
- 底部：快捷选项区（A/B/C按钮）+ 自定义输入框 + 常驻按钮（状态/背包/关系/重开）

**每回合AI输出规范（强制结构，便于解析）**

- `narrative`：剧情文本（1~4段）
- `state_delta`：属性变动摘要（可为空）
- `options`：数组（2~4个），每项含 `id/title/desc/risk_hint`
- `ask`：若需要澄清，则 options 为空，ask 为问题

**用户输入类型**

- quick：点选 option_id
- free：输入行动文本（长度上限 120 字）

**回合流程（研发实现建议）**

1. 客户端 `POST /turn`（run_id + input）
2. 服务端：校验 → 读取state → 调用剧情生成 → 状态更新 → 返回 AI消息 + options
3. 客户端渲染消息与按钮

**卡死防护**

- 若 AI 返回无 options 且无 ask：服务端兜底补“继续/查看状态/随机行动”3选项

------

### 3.3 结算 Settlement

**展示**

- 结局标题（AI生成短句）
- 死因/结局描述
- 战报：存活年限、财富峰值、声望峰值、关键关系、关键选择（3~6条）
- 奖励：记忆碎片、解锁天赋/成就

**操作**

- 立即重生 → Talent
- 分享战报（复制纯文本 + 小程序转发卡片）
- 返回首页

------

### 3.4 天赋选择 Talent

**规则（MVP）**

- 每次重生：提供候选天赋 5 个，从“已解锁池 + 随机池”抽取
- 可选 3 个（容量=3），部分天赋占用2格（稀有）
- 确认后写入 run 的 `selected_talents`

**刷新（可选P1）**

- 消耗记忆碎片刷新候选

------

### 3.5 图鉴 Atlas / 成就 Achievements / 设置 Settings

- 图鉴：已解锁显示详情；未解锁显示剪影+提示条件
- 成就：已完成/未完成 + 进度条（若可计算）
- 设置：状态简报开关、文本密度（短/中/长）、风格偏好（搞笑/严肃/爽文）

## 4. 核心玩法参数（研发&内容共同对齐）

### 4.1 状态字段（建议固定 schema）

**基础属性（0~100）**

- `int` 智力、`str` 体质、`cha` 魅力、`money` 财富（可用数值档）、`luck` 幸运

**扩展**

- `age` 年龄（整数，事件可 +0~+10）
- `risk` 死亡风险（0~100，>=100强制触发死亡事件）
- `inventory` 道具数组
- `relations` NPC关系（key-value）
- `flags` 关键事件标记（布尔或计数）
- `main_quest_stage` 主线阶段（0..n）

### 4.2 一局节奏（内容编排标准）

- 3分钟档：8~12回合，年龄跃迁更快，事件强刺激，易死
- 6分钟档（默认）：12~~18回合，1~~2条小主线
- 10分钟档：18~28回合，可触发隐藏结局（P1）

MVP默认 6分钟档：每回合事件文本 200~450字（含选项前不超过450字），避免刷屏。

## 5. AI剧情引擎设计（研发可落地）

### 5.1 架构建议（“规划器+写手”两段式，降低胡编）

- **Planner（规划器）**：根据 state + 事件卡约束，产出结构化“事件结果”（含数值变化、flag变化、死亡判定）
- **Writer（写手）**：将结构化结果改写成好看的叙事文本与选项文案

MVP可先合并为单次调用，但仍要求模型按结构化 JSON 输出，服务端校验后再渲染。

### 5.2 工具化/函数调用（推荐）

实现一组 server-side tools（即便不用模型函数调用，也按此逻辑组织）：

- `pick_event(state) -> event_card`
- `apply_choice(state, choice) -> state_delta`
- `check_death(state) -> death_card?`
- `summarize_memory(chat_history) -> summary`

### 5.3 Prompt 模板（可直接给研发）

#### System（固定）

你是“重生一亿次”文字人生模拟引擎。你必须遵守：

1. 输出必须是**严格JSON**，不得包含JSON外文本。
2. 必须基于输入state与world_rules，不得凭空改变已发生事实。
3. 若用户行动不可执行，返回 `need_clarify=true` 或提供可执行替代选项。
4. 控制篇幅：narrative不超过450中文字符。
5. 安全合规：避免血腥细节、色情细节、仇恨与违法教程；必要时用“淡化叙述+替代事件”。

#### Developer（每次请求带上世界观规则摘要）

- world_id、世界观风格（搞笑/严肃/爽文）
- 允许的超自然程度（0=现实，1=轻奇幻，2=高魔）
- 可用职业/身份列表
- 事件卡池标签与权重

#### User（每回合请求体建议）

包含：`state`、`memory_summary`、`last_ai_message`、`user_input`（option_id或free_text）、`turn_index`

#### 期望模型输出 JSON Schema（MVP）

```
json复制代码{
  "need_clarify": false,
  "ask": "",
  "narrative": "……",
  "state_delta": {
    "age_add": 1,
    "int_add": 0,
    "str_add": -2,
    "cha_add": 1,
    "money_add": 10,
    "luck_add": 0,
    "risk_add": 8,
    "inventory_add": ["旧钥匙"],
    "inventory_remove": [],
    "relations_delta": {"母亲": 2},
    "flags_set": ["met_teacher"],
    "flags_unset": [],
    "main_quest_stage_set": 1
  },
  "options": [
    {"id":"A","title":"……","desc":"……","risk_hint":"低"},
    {"id":"B","title":"……","desc":"……","risk_hint":"中"},
    {"id":"C","title":"……","desc":"……","risk_hint":"高"}

    "is_dead": false,

```

### 5.4 记忆与一致性（强烈建议实现“摘要记忆”）

- 保存最近 N=20 条对话原文 + 一个 `memory_summary`（<=800字）
- 每 5 回合触发 `summarize_memory`，将关键事实写入 summary（身份、地点、重要NPC、获得道具、主线阶段、近期矛盾）
- 模型输入永远带 `state + memory_summary`，减少跑偏

## 6. 内容生产体系（内容团队可执行）

### 6.1 内容资产类型

1. **世界观包 World Pack**

- 世界观简介（200字）
- 规则：超自然等级、职业身份、社会结构、禁用题材
- 事件标签：学业/职场/犯罪/情感/疾病/意外/奇遇…
- 主线骨架（可选）：3段式（起因-发展-高潮/结局）

1. **事件卡 Event Card（推荐做“半结构化”，AI填空）**

- `event_id`
- `trigger_conditions`：年龄范围、属性阈值、flag要求、风险区间
- `tone`：搞笑/严肃/爽文
- `prompt_stub`：给AI的写作引导（人物/冲突/可选方案）
- `outcome_templates`：每个选项的倾向（例如A偏保守+收益小，C高风险高回报）
- `death_links`：可能引到哪些死法/结局

1. **死法卡 Death Card**

- `death_id`、短标题、淡化描述模板、结算文案钩子（梗）

1. **天赋/成就库**

- 结构化：id、名称、稀有度、效果、解锁条件、占用容量

### 6.2 世界观包（MVP给内容的标准模板）

**世界观：现实·普通人生（默认）**

- 超自然：0
- 风格：可切搞笑/爽文
- 主线骨架：出身→学业/打工→关键转折（创业/婚姻/事故）→晚年/意外
- 禁止：详细暴力血腥、违法教程

> MVP先做1个世界观包 + 120张事件卡（够支撑前1-3天复玩）。后续按包扩展。

### 6.3 事件卡数量建议（MVP）

- 0-12岁：20张（家庭/学校/意外）
- 13-22岁：30张（学业/友情/恋爱/竞赛）
- 23-40岁：40张（职场/婚姻/创业/骗局/健康）
- 41-80岁：20张（健康/家庭/财产/意外）
- 死亡/结局：10-20张（含隐藏结局P1）

### 6.4 内容QA规则（必须）

- 不出现“你已经死了但还在行动”的矛盾
- 不出现“无缘由财富/身份瞬间跃迁”（除非天赋/事件解释）
- 同一NPC姓名/关系一致（使用 relations 表）
- 单回合不引入超过2个新名词/新人物（防信息过载）
- “高风险选项”必须真的更容易死/更大波动（让玩家学会权衡）

### 6.5 选项文案规范（避免模型发散）

每回合提供2-4个选项，覆盖：

- 稳健：低风险低收益
- 进取：中风险中收益
- 赌命：高风险高收益/强剧情推进
- （可选）情绪化/搞笑：带梗但可执行

### 6.6 合规与降级写作

- 暴力：只写“你受伤/失去意识/抢救无效”，避免细节
- 色情：不写露骨内容，关系推进用“克制”叙事
- 违法：不提供可操作步骤，改为“你意识到风险选择退出/被劝阻”
- 若用户强行输入违规：返回 `need_clarify=true` 并提供合规替代选项

## 7. 数据结构与接口（研发可直接设计DB/API）

### 7.1 数据库表（建议）

**users**

- `user_id` (pk), `created_at`, `channel`, `memory_balance`, `rebirth_count`

**runs**

- `run_id` (pk), `user_id`, `world_id`, `status`(active/ended)
- `state_json`（当前状态）
- `memory_summary`
- `turn_index`
- `created_at`, `updated_at`

**run_messages**（可选；也可只存摘要+最近N条）

- `run_id`, `idx`, `role`(ai/user), `content`, `created_at`

**talents**

- `talent_id`, `name`, `rarity`, `capacity_cost`, `effect_json`, `unlock_condition_json`

**user_talents**

- `user_id`, `talent_id`, `unlocked_at`

**achievements / user_achievements**

- 同理

**event_cards**（内容后台/配置）

- `event_id`, `world_id`, `trigger_json`, `prompt_stub`, `weight`, `tags`

### 7.2 核心接口（REST示例）

- `POST /v1/run/new` -> {run_id, init_ai_message}
- `POST /v1/run/continue` -> {run_id, messages, options, state_brief}
- `POST /v1/run/turn` (run_id, input_type, option_id/free_text) -> {ai_message, options, state_brief, death?}
- `POST /v1/run/settle` (run_id) -> settlement payload（若turn已判定死亡可省）
- `GET /v1/talents/candidates?user_id=...` -> 候选天赋
- `POST /v1/talents/select` (run_id, talent_ids) -> ok + next_run_init
- `GET /v1/atlas` / `GET /v1/achievements`
- `POST /v1/share/text` -> 生成分享文案（也可前端拼）

## 13. 技术实现要点（研发注意）

### 13.1 选项按钮与自由输入的统一处理

- 客户端统一发送 `input_type`
- 服务端将 option 映射为明确意图文本（避免模型误解），例如：
  - option A 内部意图：“请求调监控，提供证据，保持礼貌”
  - free_text 原样传入，但做敏感词过滤

### 13.2 规则引擎（最小可用）

- 数值 clamp：属性0-100，risk 0-120（>100即死）
- 天赋效果：至少支持
  - 开局加成
  - 结算碎片倍率
  - 免死一次（per run）
- 死亡判定：risk>=100 或 death_card命中

### 13.3 降级策略（稳定性）

- 模型超时：返回“系统走神了”+ 3个保底选项（继续/查看状态/随机行动），并记录 `error_occurred`
- 结构化JSON解析失败：自动重试一次（带更严格提示），仍失败则走降级文案
